全体構成

現場の設備からダッシュボードまで、データは上から下へ(実線)、制御は右側を通って 下から上へ(緑の破線)流れます。

データフロー: 収集から可視化まで

  1. デバイスやゲートウェイが、gRPC(GatewayIngress)・MQTT・AMQP のいずれかでテレメトリを送信します。
  2. ConnectorWorker が JSON Schema でペイロードを検証し、デジタルツインの情報(建物・機器・ポイント名)を付与します。
  3. 検証済みデータは NATS JetStream の building-os.validated.telemetry に集約されます。
  4. 最新値は NATS KV(Hot)へ、履歴は ParquetLakeWriter が MinIO 上の Parquet レイク(Warm / Cold)へ書き込みます。
  5. API の GET /telemetries/query が階層を自動選択して返し、Next.js ダッシュボードが表示します。

制御フロー: API から現場へ

POST /points/{id}/control を受けた API は 202 Accepted と controlId を返し、制御要求を NATS のゲートウェイ別サブジェクト control.request.gw.{id} に送ります。GatewayBridge が gRPC の双方向ストリームで ゲートウェイに届け、ゲートウェイが BACnet などの現場プロトコルで設備に書き込みます。

対象のゲートウェイがオフラインの場合はタイムアウトを待たずに即時 503 を返します(fast-fail)。 すべての制御操作は PostgreSQL の監査テーブル(point_control_audit)に記録され、 「いつ・誰が・どのポイントに・何を書いたか」を後から追跡できます。

テレメトリの 3 階層

階層置き場所持つもの主な用途
HotNATS KVポイントごとの最新値ダッシュボードのリアルタイム表示
WarmMinIO Parquet レイク直近の履歴・集計トレンド分析・レポート
ColdMinIO Parquet レイク長期アーカイブ長期分析・研究データ

利用者が階層を意識する必要はありません。単一の API GET /telemetries/query が、 指定された期間と粒度から適切な階層を自動選択します。詳細は oss-tier-architecture.md / oss-warm-parquet-lake.md を参照してください。

デジタルツインと標準オントロジー

建物→フロア→空間→機器→ポイントの階層は、OxiGraph(RDF / SPARQL)上のデジタルツインとして 管理されます。主語彙は SBCO オントロジーで、これがシステム全体の ポイントリストの正本です。主要な建物オントロジー標準への対応表も整備しています。

SBCOBrickRECIFCDTDL
sbco:Buildingbrick:Buildingrec:BuildingIfcBuilding…:Building;1

※ 対応表の全体(フロア・空間・機器・ポイント含む)は standard-mapping.md を参照。

デプロイ

  • Docker Compose

    ローカル開発・小規模検証向け。docker-compose.oss.yaml 1 ファイルで全サービスが起動します。

  • Helm + Kubernetes

    本番運用向け。各コンポーネントの Helm チャートを同梱し、水平スケールに対応します。

  • Argo CD + OpenTofu

    GitOps による継続的デリバリと、OpenTofu によるインフラのコード管理を想定した構成例を提供します。

→ oss-production-deployment.md ・ argocd-gitops-guide.md
