# フロントエンド通知・エラー表示ポリシー（#162）

Web Client のエラー／成功の**出し方を統一**するための指針。再評価 §4 P1-4（エラー表示が画面ごとに
バラバラ）への回答。部品は**依存ライブラリなし**で `components/ui` に集約する。

## 2 つの部品と使い分け

| 種別 | 部品 | いつ使う | 挙動 |
|---|---|---|---|
| **トースト** | `useToast().showToast(msg, { tone })`（`components/ui/toast.tsx`）| **一過性**のフィードバック — 操作の**成功**、一過性の通信エラー | 画面右下に浮かび**自動消滅**（既定 5s）。`error` は `role="alert"`、他は `role="status"` |
| **インラインバナー** | `<InlineBanner tone>`（`components/ui/inline-banner.tsx`, #196）| **消えると困る説明** — 取得失敗、権限不足、バリデーション、gateway offline | in-flow で**永続表示**（手動 dismiss）。`error` は `role="alert"` |

原則: **「操作の結果を一言伝えて消えてよい」= トースト**、**「なぜダメか・何を直すかを読ませたい」=
インライン**。モーダル内のエラーはそのモーダル内にインライン表示（トーストは画面全体スコープ）。

## エラー種別 → 表示（既定）

| エラー種別 | 表示 | 文言の方針 |
|---|---|---|
| 操作成功（保存 / 再同期 / 制御成功）| トースト（success）| 何が起きたかを1文（例:「再同期を通知しました」）|
| 一過性の通信エラー（リトライで直る類）| トースト（error）| 「通信に失敗しました。時間をおいて再試行してください」|
| データ取得失敗（画面が描けない）| インライン（error）| 「〜の取得に失敗しました。」+ 可能なら再読込導線 |
| 権限不足（403）| インライン（error/warn）| **必要な権限を明示**（例:「この操作には `point:{id}:write` 権限が必要です」）|
| バリデーション | インライン（error）+ 該当フィールド近傍 | 何をどう直すか |
| gateway offline（503, #186）/ タイムアウト | インライン（該当操作の近傍）| 状態と次アクション（デバイス直接確認など）|

## 既存の適用例（リファレンス）

- **制御の 403 / 503**: `point-control-modal` → `ControlStatusBar` にインライン表示（#162 / PR #227）。
- **取得失敗**: 各 `page-component` の `InlineBanner tone="error"`（#196）。
- **再同期の成功**: `/admin/gateways` の resync 成功はトースト（success）、失敗はインライン（本ポリシー適用）。

## 適用の広げ方

新規 UI は上表に従う。既存画面の移行は段階的に行い、`console.error` へ落ちている失敗を上記のどちらかに
必ず載せる（沈黙させない）。トーストは画面全体スコープなので `AppShell` に一度だけ `ToastProvider` を
マウント済み。
