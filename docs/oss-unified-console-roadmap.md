# 統合管理コンソール + 解説/オンボーディング強化 ロードマップ

## 背景と目的

現状、UI は用途別に分かれている。

| UI | オーディエンス | 権限モデル |
|---|---|---|
| `web-client` (3000) | 建物管理者 | リソース単位 (`building:id:action`) |
| `admin-console` (3001) | 管理者 | `IsAdmin` / ロール |
| 観測スタック (Grafana 等) | プラットフォーム運用者 | — |

これを次の方針で再編する。

1. **単一ウェブアプリ + ロール出し分け** — 運用者と建物管理者が同一人物のケースがあるため、ログイン先を1つにし、ロール（`building_os_role` / permissions）で見える範囲を変える。
   **鉄則: UI の出し分けは利便性であり認可ではない。API 側のエンドポイント単位認可は UI に依らず常に効かせる。**
2. **簡易モニタリング（Grafana 任意化）** — 「今動いているか・直近の数値」を見る運用トリアージ用ビューをアプリ内に持つ。深掘り（履歴/PromQL/トレース相関）は Grafana の役割として残す。**Prometheus は core（ヘッドレス）/ Grafana は optional**。
3. **コンフィグ管理（2層）** — インフラ/シークレットは「読み取り専用の実効値ビュー」（編集は IaC/ArgoCD のまま）、アプリ/ドメイン設定のみ限定的に UI 編集可（Postgres 永続）。GitOps ドリフトを起こさない。
4. **解説 / オンボーディング強化（主軸）+ 実験的 LLM（任意の薄皮）** — スコープは「ビルOS の使い方」「今の表示が何を意味するか」に限定（テレメトリ実値の問い合わせは対象外）。知識は小さく固定なので **RAG / pgvector は不要**。決定的なヘルプ UI（Tier 0）を主軸とし、同じ解説コンテンツをプロンプトに注入する軽量 LLM チャット（Tier 1）は任意プロファイルの実験機能とする。

## スコープ外（明示）

- テレメトリ中身の自然言語問い合わせ / データ分析エージェント
- pgvector / 埋め込みパイプライン / ベクタ検索
- LLM 経由のデバイス制御（制御は常に human-in-the-loop + 既存 writable/権限ゲート）
- インフラ/シークレット設定の UI からの書き換え（IaC/ArgoCD が source of truth）

## エピックとイシュー

### Epic A: 統合フロントエンド（単一アプリ + ロール出し分け）

- **A-1**: 統合アプリ基盤 — route groups（`(operator)`/`(admin)`/`(platform)`）、ロール判定ユーティリティ、ナビ出し分け、Keycloak 単一クライアント化。サーバーコンポーネント側ルートガード必須。
- **A-2**: 共通アプリシェル + ワークスペース切替 UI（複数ロール保持時の「今どの帽子か」を明示）。
- **A-3**: 管理機能（users/groups）の統合アプリへの移植。

### Epic B: 簡易モニタリング（Grafana 任意）

- **B-1**: `GET /api/system/status` 集約エンドポイント — 各サービスの up/down（`/health` 横断・Prometheus 非依存）+ 主要 KPI（msg rate / control req など）。KPI のみ Prometheus HTTP API 経由で取得し、未配線時は null に degrade（Grafana 非依存）。**TDD の最初の一歩・他機能の土台**。
- **B-2**: ConnectorWorker / ConsumerWorker に liveness/readiness エンドポイント追加（横断 health のため）。
- **B-3**: platform ワークスペースの status ダッシュボード画面（直近5〜15分・KPI 5個程度・誰に連絡するか）。

### Epic C: コンフィグ管理（2層）

- **C-1**: 実効設定ビュー — 読み取り専用 API + 画面。シークレットは必ずマスク、`IsAdmin` ゲート。
- **C-2**: アプリ設定ストア — Postgres の config テーブル + `SystemConfigController`。フィーチャーフラグ/閾値など GitOps と衝突しない設定のみ限定編集。出所（UI/IaC）を記録。

### Epic D: 解説 / オンボーディング（+ 実験的 LLM）

- **D-1**: 解説コンテンツ基盤 — content-as-code（リポジトリ内 Markdown/JSON, i18n=日本語）、用語集（`bos:` 用語・各メトリクスの意味）、画面コンポーネントに紐づくヘルプドロワー/ツールチップ。**ここで書くコンテンツが Tier 1 LLM のプロンプト原資になる（source of truth を1つに）**。
- **D-2**: オンボーディングツアー（初回ガイド）。
- **D-3**: （実験的・任意）Ollama 任意プロファイル + ヘルプ Q&A チャット。D-1 の解説コンテンツをプロンプト注入（RAG なし）。read-only・画面コンテキストのみ。`docker compose --profile assistant`。

## 依存関係と推奨順序

```
B-1 (system/status API)  ──┬─> B-3 (status 画面)
                            └─> D-3 (LLM の最重要ツール: 任意)
B-2 (worker health) ──────────> B-1 を充実
A-1 (統合基盤) ──> A-2 ──> A-3
                  └──────> B-3 / C-1 / D-1（各ワークスペース画面の置き場）
D-1 (解説基盤) ──> D-2 / D-3
```

推奨着手順:
1. **B-1**（backend・TDD しやすく土台になる）
2. **A-1**（統合シェル基盤）
3. B-2 → B-3（モニタリングを縦に通す）
4. D-1 → D-2（解説/オンボーディングの主軸）
5. C-1 →（必要なら）C-2
6. （余力で）D-3 実験的 LLM

## 開発プロセス

- 画面デザイン → テストドリブン（TDD）で実装。
- **イシュー毎に専門エージェントがコードレビュー**を実施。
- 適当な粒度で PR を発行（`feat(#NNN):` / `fix(#NNN):`）。
- **PR は自動マージしない** — CI 結果を報告し、マージ判断はユーザーに委ねる。
