# 同時送信数制限機能

## 概要

1000台以上の大規模デバイス負荷試験において、単一のIoT Hub接続を使用する場合にメッセージロスが発生する問題を解決するため、同時送信数を制限する機能を実装しました。

## 問題の背景

### 発生していた問題

- 300-600台: 基本的に正常動作するが、30分に数回程度メッセージが0件になる
- 1000台以上: 不安定になり、メッセージロスが頻発する

### 原因

1. **メッセージ送信の集中**: すべての仮想デバイスが同じタイミングでメッセージを送信
2. **SDKバッファの溢れ**: Azure IoT Device SDKの内部バッファが1000台分の同時送信に対応できない
3. **IoT Hubのスロットリング**: 瞬間的な大量メッセージでレート制限に引っかかる

## 実装した解決策

### 1. 初期遅延によるメッセージ分散

各仮想デバイスの起動時にランダムな遅延を追加し、メッセージ送信タイミングを分散。

**実装箇所**: `VirtualDevice.run_continuous()`
- `initial_delay_max`パラメータで最大遅延時間を指定
- デフォルトではメッセージ送信間隔（60秒）の範囲でランダムに遅延

### 2. セマフォによる同時送信数制限

`asyncio.Semaphore`を使用して、同時にメッセージを送信できるデバイス数を制限。

**実装箇所**: `SharedConnectionDevice`
- `max_concurrent_sends`パラメータで同時送信数を設定
- デフォルト: 50（環境に応じて調整可能）

## 設定方法

### config.jsonでの設定

```json
{
  "scenario_name": "デバイススケーリング負荷試験",
  "scenario_type": "device_scaling",
  "max_concurrent_sends": 100,
  ...
}
```

### パラメータ調整ガイド

#### `max_concurrent_sends`の推奨値

| デバイス数 | 推奨値 | 説明 |
|-----------|--------|------|
| ~300台 | 30-50 | デフォルト設定で十分 |
| 300-1000台 | 50-100 | 中規模負荷に対応 |
| 1000-2000台 | 100-200 | 大規模負荷に対応 |
| 2000台~ | 200-300 | 超大規模負荷（IoT Hub SKUに注意） |

**注意**: 
- 値を大きくしすぎると、SDKバッファが溢れる可能性があります
- 値を小さくしすぎると、スループットが低下します
- IoT HubのSKU（S1/S2/S3）によって最適値が異なります

## 動作の仕組み

### メッセージ送信フロー

```
1. 仮想デバイス起動
   ↓
2. 初期遅延（0～60秒のランダム）
   ↓
3. メッセージ送信ループ開始
   ↓
4. セマフォ取得待機（同時送信数が max_concurrent_sends 未満になるまで待つ）
   ↓
5. メッセージ送信
   ↓
6. セマフォ解放
   ↓
7. 60秒待機
   ↓
8. 3に戻る
```

### 具体例（1000台の場合）

- **初期遅延あり**: 60秒間に1000台が均等に分散して送信開始
- **セマフォ制限（100）**: 同時に最大100台のみが送信処理を実行
- **結果**: 1000台のメッセージが約10-15秒かけて順次送信される

## 統計情報

実装により、以下の統計情報が追加されました：

- `throttled_counts`: スロットリングされた回数
- `throttle_rate`: スロットリング率

これらの情報は`SharedConnectionDevice.get_stats()`で取得できます。

## トラブルシューティング

### それでもメッセージロスが発生する場合

1. **`max_concurrent_sends`を減らす**
   - 現在の値から50%程度減らしてみる
   - 例: 100 → 50

2. **IoT HubのSKUを確認**
   - S1: 400メッセージ/秒
   - S2: 6,000メッセージ/秒
   - S3: 300,000メッセージ/秒

3. **ログを確認**
   - `throttled`エラーが多発していないか
   - 接続エラーが発生していないか

4. **Azure Monitorで確認**
   - IoT Hubメトリクスでスロットリング状況を確認
   - ルーティング遅延を確認
